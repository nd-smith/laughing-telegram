# PropGateway — Product Requirements Document

## Context

The Property organization of a large insurance company needs a real-time event-data pipeline to move data from external source systems into centralized storage (Microsoft Fabric Lakehouse) and make it available to downstream consumers via Kafka. External sources produce differently-shaped events (primarily status updates, some document-ready notifications with pre-signed S3 URLs). The pipeline must handle multiple sources with clean separation, be simple enough for non-expert maintainers, and be optimized for AI-assisted troubleshooting.

This PRD covers the **core pipeline only**. Companion apps (business logic, enrichment, hydration, API triggers) will live in the `/apps` folder of the same monorepo but are out of scope for this document.

---

## Guiding Principles

1. **Simplicity first.** Choose the simplest solution that is good enough. Avoid abstraction layers, DSLs, and frameworks that add indirection.
2. **Optimized for AI-assisted maintenance.** Code should be readable and explicit. Maintainers are not coding professionals — Claude (or similar) will be used to diagnose and troubleshoot.
3. **Deliberate complexity.** Every piece of complexity must earn its place. If plain Python is sufficient, don't build a framework.

## System Overview

### Pipeline Flow

```
External Sources
       │
       ▼
Azure Event Hubs (one per source, external ingestion)
       │
       ▼
PropGateway Pipeline Worker (one process per source)
  ├── 1. Consume from Event Hub
  ├── 2. Validate (minimal — required fields check)
  ├── 3. Extract metadata (per-source Python helper)
  ├── 4. Create envelope (standard schema wrapping raw event)
  ├── 5. Buffer & write to OneLake (micro-batch Parquet/Delta)
  └── 6. Publish envelope to Kafka (per-source topic)
       │
       ▼
Downstream consumers (Kafka per-source topics → future /apps)
```

### Key Architecture Decisions

| Decision | Choice | Rationale |
|---|---|---|
| External ingestion | Azure Event Hubs (separate hub per source) | External sources push here; isolation per source |
| Internal messaging | Self-managed Kafka on K8s | Internal backbone; per-source topics for fan-out |
| Storage | Microsoft Fabric Lakehouse via OneLake (ADLS Gen2 API) | Already provisioned; micro-batch Parquet/Delta writes |
| Auth | Azure AD / Entra | Service identities for all Azure resource access |
| Scaling model | K8s replicas | Each worker is a single process for one source; scale by adding replicas |
| Worker selection | CLI argument at runtime | e.g., `python -m pipeline --source source-a` |
| Per-source logic | Python helper functions | Not YAML/config DSL — plain Python is simpler to read, debug, extend |
| Concurrency model | Sync + threading | Simpler than asyncio; easier for non-experts and AI to reason about |
| Cross-cloud | Yes | Some upstream sources are on AWS (pre-signed S3 URLs passed through) |

## Envelope Schema

Every event processed by the pipeline is wrapped in this envelope before storage and publishing.

```json
{
  "envelope_id": "uuid v4 — unique per envelope",
  "correlation_id": "string | null — for cross-system tracing, extracted from source event if available",
  "schema_version": "string — envelope schema version (e.g., '1.0')",
  "source": {
    "system": "string — source system identifier (matches CLI --source arg and source config)",
    "event_hub": "string — originating Event Hub name",
    "original_timestamp": "ISO8601 | null — timestamp from the source event if available"
  },
  "event": {
    "type": "string — event type (e.g., 'status_update', 'document_ready')",
    "received_at": "ISO8601 — when the pipeline received the event from Event Hub",
    "processed_at": "ISO8601 — when the pipeline finished processing"
  },
  "metadata": {
    "// Per-source extracted fields, defined by each source's Python extractor": "",
    "// Examples: policy_id, claim_number, document_url, status, etc.": ""
  },
  "pipeline": {
    "pipeline_id": "string — identifier for this pipeline instance/worker",
    "retry_count": "int — number of processing retries (0 on first attempt)",
    "processing_ms": "int — total processing time in milliseconds"
  },
  "payload": {
    "// Original raw event body, untouched": ""
  }
}
```

### Field Requirements

- `envelope_id`: Always generated by pipeline. UUID v4.
- `correlation_id`: Extracted from source event if present, otherwise null.
- `schema_version`: Hardcoded in pipeline code. Bumped on breaking envelope changes.
- `source.system`: Must match a registered source name.
- `source.event_hub`: Populated from Event Hub connection metadata.
- `source.original_timestamp`: Best-effort extraction from source event.
- `event.type`: Extracted by per-source Python helper.
- `event.received_at`: Set when event is read from Event Hub.
- `event.processed_at`: Set after envelope is fully constructed.
- `metadata`: Variable shape per source. Defined by each source's extractor function.
- `pipeline.pipeline_id`: Set from environment/config at worker startup.
- `pipeline.retry_count`: Incremented on each retry.
- `pipeline.processing_ms`: Wall-clock time from receive to envelope completion.
- `payload`: The original event body, preserved exactly as received.

## Per-Source Configuration

Each source is a Python module in `pipeline/sources/`. A source module defines:

1. **Connection config** — Event Hub connection string/reference, Kafka output topic name
2. **Extractor function** — Takes the raw event, returns extracted metadata dict and event type

Example source module (`pipeline/sources/source_a.py`):

```python
"""Source configuration and extractor for Source A (policy status system)."""

SOURCE_NAME = "source-a"
EVENT_HUB_NAMESPACE = "prop-eventhub-ns"
EVENT_HUB_NAME = "source-a-events"
KAFKA_OUTPUT_TOPIC = "propgateway.source-a"

def extract(raw_event: dict) -> tuple[dict, str]:
    """Extract metadata and event type from a raw Source A event.

    Args:
        raw_event: The parsed JSON body of the incoming event.

    Returns:
        Tuple of (metadata_dict, event_type_string).
    """
    return (
        {
            "policy_id": raw_event["data"]["policy"]["id"],
            "status": raw_event["data"]["status"],
        },
        raw_event.get("type", "status_update"),
    )
```

**Adding a new source** = adding a new Python file in `pipeline/sources/` with these same exports. No framework registration, no config files to wire up.

## Pipeline Components

### 1. Entry Point (`pipeline/__main__.py`)

- Parses CLI arguments: `--source <source-name>` (required)
- Loads the matching source module from `pipeline/sources/`
- Initializes consumer, writer, publisher
- Runs the main processing loop
- Handles graceful shutdown (SIGTERM/SIGINT)

### 2. Consumer (`pipeline/consumer.py`)

- Connects to Azure Event Hub using `azure-eventhub` SDK
- Authenticates via Azure AD / Entra (`azure-identity` — DefaultAzureCredential)
- Reads events in batches
- Passes each event to the processing pipeline
- Manages checkpointing (tracks which events have been processed)

### 3. Validation (`pipeline/validation.py`)

- Minimal validation — checks that the raw event:
  - Is valid JSON
  - Contains expected top-level fields (per source extractor requirements)
- Returns pass/fail with reason
- Failed events are routed to retry/DLQ flow

### 4. Envelope (`pipeline/envelope.py`)

- Takes raw event + extracted metadata + event type
- Constructs the full envelope dict per the schema above
- Generates `envelope_id` (UUID v4)
- Sets timestamps (`received_at`, `processed_at`)
- Calculates `processing_ms`

### 5. Writer (`pipeline/writer.py`)

- Buffers enveloped events in memory
- Flushes to OneLake as Parquet files on interval (configurable, default 60 seconds) or when buffer reaches size threshold
- Writes via `azure-storage-file-datalake` SDK (ADLS Gen2 API)
- Uses `pyarrow` for Parquet serialization
- File path convention: `{lakehouse_path}/{source}/{year}/{month}/{day}/{timestamp}_{batch_id}.parquet`
- Authenticates via Azure AD / Entra

### 6. Publisher (`pipeline/publisher.py`)

- Publishes enveloped events to the source's Kafka output topic
- Uses `confluent-kafka` producer
- Serializes envelope as JSON
- Handles delivery confirmation callbacks
- Failed publishes trigger retry/DLQ flow

## Error Handling

### Retry Strategy

- On processing failure (validation, write, publish): retry up to **3 times** with exponential backoff
- `pipeline.retry_count` in the envelope tracks attempts
- After max retries exhausted: send to dead letter queue

### Dead Letter Queue

- DLQ is a dedicated Kafka topic per source: `propgateway.{source-name}.dlq`
- DLQ messages include:
  - The original raw event
  - The error message and stack trace
  - Retry count
  - Timestamp of final failure
- DLQ consumers (monitoring/alerting) are out of scope for this PRD

## Observability

### Structured Logging

- JSON-formatted logs to stdout (K8s collects from stdout)
- Every log line includes: `timestamp`, `source`, `correlation_id`, `envelope_id` (when available), `level`
- Log levels: DEBUG, INFO, WARNING, ERROR
- Key events logged: event received, validation pass/fail, envelope created, batch written, Kafka published, retry, DLQ

### Metrics

- Prometheus-compatible metrics exposed via HTTP endpoint (for K8s scraping)
- Key metrics:
  - `events_received_total` (counter, by source)
  - `events_processed_total` (counter, by source, status=success|failure)
  - `events_dlq_total` (counter, by source)
  - `processing_duration_seconds` (histogram, by source)
  - `batch_write_duration_seconds` (histogram, by source)
  - `batch_size` (histogram, by source)
  - `kafka_publish_duration_seconds` (histogram, by source)
  - `buffer_size` (gauge, by source — current events in write buffer)

## Scale & Performance

- **Target throughput:** Thousands of events per minute per source
- **Write buffer flush:** Every 60 seconds or 1000 events (whichever comes first), configurable
- **Scaling:** Horizontal via K8s replicas — multiple workers for the same source share Event Hub consumer group partitions
- **Concurrency:** Sync + threading within a single worker process

---

## Authentication & Security

- All Azure resource access via **Azure AD / Entra** using `DefaultAzureCredential`
- No API keys or connection strings in code — managed identities or environment-based credentials
- Pre-signed S3 URLs from AWS sources are **passed through** in the envelope (pipeline does not fetch documents)
- Network-level security handled by K8s/infrastructure (out of scope)

---

## Success Criteria

1. Pipeline successfully consumes events from an Azure Event Hub source
2. Events are validated, enveloped, and written to OneLake as Parquet files
3. Enveloped events are published to the correct per-source Kafka topic
4. Failed events (after retries) land in the source's DLQ Kafka topic
5. Metrics are exposed and scrapable by Prometheus
6. A new source can be added by creating a single Python file in `pipeline/sources/`
7. Unit tests pass for all core components
